# Mindset

- Understand goal, constraints, and current behavior before editing.
- **Tests first for refactoring**: Before making changes to existing code, ensure tests exist that document current behavior. Write tests if they don't exist.
- **TDD for new features**: When implementing new features, write tests before implementation.
- If the task is ambiguous, high-impact, or touches public APIs, schemas, auth, or infra, ask clarifying questions before changes.
- Do not ask for confirmation for obviously safe read-only actions (reading files, listing dirs, simple searches).
- If confidence is low, say so and propose a safe next step instead of guessing.
- Prefer asking over assuming - clarify before making assumptions about requirements.
